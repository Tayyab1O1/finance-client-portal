# Standing Rules

_Durable conventions for this project — read this before touching the AP/AR dashboards, Fillout integration, or role system. `PROGRESS.md` is the changelog of what's been built; this is the "don't relitigate these decisions" file._

## Admin must always have full power to mimic any client or bookkeeper

Admin is the ceiling, not just another role. Concretely:

- **Every tab/section in `ClientPortalView` must be visible to admin regardless of any client-facing enable toggle** (`formsEnabled`, `dashboardsEnabled`, etc.). Those toggles control what the *client* (and bookkeeper) sees — never what admin can inspect or manage. Enforced in `ClientPortalView.tsx` via an `isAdmin` bypass in the tab filter — if a new tab/section is added later, it must respect this same bypass.
- **Admin can write every recognized transaction field** — every structural field (`canWriteField`) and every admin-defined `extraFields.<id>` regardless of its configured `editableBy` (`canWriteExtraField`'s admin branch always returns true). The one deliberate exception is `formAnswers.*` (the submitted form data), which is immutable for every role including admin — see "Form-submitted data is immutable" below. That's a considered exception, not a gap in the admin-ceiling rule.
- **Admin can read every client's data, always** — enforced in `firestore.rules` (`isAdmin()` short-circuits every `canViewClient()` check before the client/bookkeeper-scoped conditions are even evaluated).
- When adding any new gated feature (a new toggle, a new role-scoped view, a new field), ask: "would this hide something from admin that a client or bookkeeper can see or do?" If yes, add the admin bypass before shipping it, not after someone notices it's missing.

## Secrets

- **Never pipe a secret through `grep`/`cat` into `firebase functions:secrets:set --data-file -`.** A raw pipe preserves its own trailing newline, which gets baked into the stored secret and silently breaks every exact-match comparison downstream (this happened once — see `PROGRESS.md`, cost a long debugging session). Always use command substitution instead: `printf '%s' "$VALUE" | firebase functions:secrets:set NAME --data-file -`.
- The same secret values must be set in **two places** for Fillout: Firebase Secret Manager (for `functions/`) and the Next.js app's server env (`client-portal/.env.local`, for the `/api/admin/fillout/*` and `/api/transactions/write-field` routes). They are not automatically kept in sync — changing one without the other reintroduces exactly the class of bug above.

## Deploying Firebase Functions

- **`npx tsc --noEmit` does NOT emit compiled output.** It only type-checks. Deploying without first running `npm run build` (inside `functions/`) silently re-uploads whatever was last compiled — which can be arbitrarily stale relative to `src/`. This happened once (see `PROGRESS.md`) and produced a confusing hour of "the code should be doing X but isn't." Always run `npm run build` immediately before `firebase deploy --only functions[:name]`, and sanity-check `lib/index.js`'s mtime is newer than `src/index.ts`'s if anything looks off.
- Deploying Firestore rules similarly requires the real deploy, not just `--dry-run`. `--dry-run` only validates compilation; it does not release the rules. This also happened once and produced "Missing or insufficient permissions" errors despite the rules file itself being correct.

## Identity and trust boundaries

- **Never trust a payload for its own identity.** The Fillout webhook's client/dashboard identity comes from the `?formId=` query param registered at connect-time (looked up against `filloutFormMappings`), never from anything inside the request body. The write-field endpoint derives a transaction row's `clientId` by reading the row from Firestore server-side, never from a client-supplied value.
- **Field-level write permission lives in exactly one place**: `lib/transactionFieldOwnership.ts`. Structural fields (`apRunDate`, `attachmentLinks`) resolve statically via `getFieldOwner`/`roleCanWriteField`/`canWriteField`. Admin-defined fields (`extraFields.<id>`) resolve dynamically via `canWriteExtraField()`, given that field's actual `ExtraFieldDef` (whose `editableBy` admin configured through `DashboardFieldsEditor`, stored in `dashboardFieldConfigs`) — never a static list, since admin authors these per client+dashboard. Both the grid UI and the `/api/transactions/write-field` route call the same functions in this one file for both cases. If a permission bug surfaces, fix it here — don't patch around it in the UI or the route individually, or the two will drift and one will silently disagree with the other.
- **Form-submitted data is immutable — for every role, including admin.** `formAnswers.*` deliberately returns `null` from `getFieldOwner()`, so the write-field route rejects any attempt to edit it (400, "not editable through this endpoint") and the grid renders those cells as plain read-only text. This was an explicit reversal of an earlier decision (form answers were briefly bookkeeper-editable, for e.g. GL-code corrections) — the submitted answer now stays a fixed record of what was actually sent; corrections happen via an admin-defined `extraFields` column instead (e.g. a "Corrected GL Code" field), never by overwriting the original. Don't reopen `formAnswers.*` editing without checking this was actually reversed on purpose.
- Direct Firestore writes to `apTransactions`/`arTransactions`/`formSchemas`/`filloutFormMappings` are blocked at the rules layer (`allow write: if false`) for every role including admin's own browser session — all writes to these collections go through server routes/functions using the Admin SDK, which is what actually enforces the field-ownership rules above. Don't "simplify" this by opening direct client writes even for a quick admin-only feature. (`dashboardFieldConfigs` is the one exception — admin writes it directly, since defining a field has no external side effect the way connecting a Fillout webhook does.)

## Data model conventions

- Form-sourced fields live in an open `formAnswers: { [questionId]: value }` map, keyed by Fillout's stable question `id` (not the label). A form question's label can be edited freely without any migration; only deleting-and-recreating a question mints a new `id` and behaves like an unrelated new column.
- Manual/status fields live in an equally open `extraFields: { [fieldId]: value }` map — there is no fixed set of these baked into `TransactionRow`. Admin defines the field list per client+dashboard (`dashboardFieldConfigs`); the grid renders whatever's there, in whatever order admin set. Don't reintroduce fixed fields like `vendorName`/`amount`/`clientStatus`/`apRunDate` as first-class `TransactionRow` properties — that's the exact design this was changed away from. AP-run-date is now an opt-in `extraFields` entry too (`type: "apDate"`), not an automatic column — its options come from the client's `ApCalendar` (`effectiveApDates`) rather than admin-typed options, and its value is validated against that calendar server-side in `write-field/route.ts` by checking the field def's `type`, not by field name. `attachmentLinks` is the one manual/status-adjacent thing that's still genuinely structural (every row gets it, not admin-optional) since it's tied to file uploads specifically.
- `ApCalendar.generatedDates` is append-only in spirit: extending the window or changing recurrence must never drop a date already generated, because a transaction row may already be assigned to it. Use `skipDates` to exclude a specific occurrence, not by removing it from `generatedDates`.
- Transaction docs are keyed by Fillout's `submissionId` (not an autogenerated Firestore ID), making both the real-time webhook and the scheduled reconciliation job naturally idempotent — re-ingesting the same submission is always an upsert, never a duplicate.
